                                  开发规约

                                  细节决定一切，疑难杂症的问题觉得无法入手，简单的问题留有遗漏，这就是菜

                                  # 代码review

                                  1、中大型需求在提测时必须与结对编程人共同review，探讨实现细节

                                  2、业务负责人review，合并stage时，需认真查看其代码是否存在逻辑缺陷；规范缺陷

                                  # 代码规范与命名

                                  严格遵守代码规范与命名规范

                                  # 不允许携带调试内容

                                  不允许携带debugger、console到master分支，调试内容当时用完当时删

                                  # 域名收敛

                                  检查静态资源域名，图片使用s2、其他资源使用static2、cdn使用cdn.51cto.com，不允许出现其他诸如s1、static1形式的域名

                                  # 逻辑处理

                                  逻辑处理，不允许出现一部分在前端处理，一部分在后端处理

                                  # 模型驱动

                                  模型驱动，开发之前必须就数据模型与后端达成一致，不允许出现后端接口不通导致联调时间过长

                                  每一次开发前必须与后端确定数据模型，需要几个接口，接口入参，接口返回字段 没做这一步的，绩效一律5分

                                  # 接口频率

                                  接口请求标记，针对导出，搜索等按钮，限制请求频率

                                  # UI交互性

                                  UI交互，按钮等可点击模块，指向小手等UI变化。具体可以根据UI要求灵活变动，但禁止交互性内容无效果

                                  按钮颜色，如删除红色，危险操作警告色，普通按钮蓝色等

                                  # tab栏缓存

                                  缓存当前切换的tab栏

                                  # 不可逆操作提示

                                  删除等不可逆操作需确认

                                  # 尽可能的loading与提示

                                  禁止如何点击无响应，给出相应的loading或提示

                                  # 接口异常

                                  任何接口异常必须有提示（可见性的提示，不能是log）；同时针对数据做非空判断，异常判断；禁止出现任何形式的点击无响应

                                  # 精度问题

                                  js计算精度丢失问题、取整数据

                                  需与后端，产品达成一致，决定如何确定精度，禁止单方面的感觉如取两位小数，向上向下取整

                                  同时关于精度问题偏向用户层面考虑，以更宽松一些策略保证健壮性（前提如上，需要达成一致）

                                  # 图片alt

                                  所有的图片元素，必须具有alt属性，且不能为空；（改造旧代码时要特别注意，为其新增）

                                  # 禁止询问学员密码

                                  排查线上问题如果确实需要登录学员账号，可以采用手机号后台拿到验证码登录，必须让业务同事（如本人沟通则是自己）取得学员许可，并知会学员会发送验证信息，不予理会即可

                                  # 网络问题排查路径

                                  1、检查学员的本地host文件，是否正常。如果host异常，可能会影响访问。

                                  2、检查学员本地的VPN代理，

                                  3、检查学员本地的DNS，这个有wiki专门处理这个，可以自行查找

                                  4、检查浏览器版本，推荐使用chrome浏览器就行访问

                                  5、检查浏览器缓存，必要时可以清除浏览器缓存，或者把浏览器设置为不保存缓存的模式

                                  以上检查如果都排查完毕之后，还有问题，可以联系主管进行专门的会议解决
